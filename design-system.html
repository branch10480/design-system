特徴

調査結果や仕様書を HTML として公開するときの「いつもの土台」。

単一ファイルで完結

外部 CSS / JS / CDN に依存しない。フォントはシステムフォントと、リポジトリ同梱の UDEV Gothic 35LG のみ。:root のトークンブロックをコピーすれば、どの成果物も同じ見た目になる。

ライト / ダーク自動対応

色は必ずトークン経由で参照する。OS の外観設定に追従し、右上のトグルで手動切り替えもできる。

Apple Docs の語彙

swift-docc-render のカラーパレットと developer.apple.com のレイアウト語彙（eyebrow / abstract / aside / availability）を踏襲。

Foundations

すべての成果物が共有する土台。色・文字・余白・角丸・影は、ここで定義したトークンだけを参照する。

Light ✓ Dark ✓ Beta

概要

トークンは「表面（surface）」「文字（text）」「ティント（tint）」「ステータス」の 4 族。リテラルの hex をコンポーネント側に直書きせず、必ず CSS 変数経由で参照する。

重要

ダーク対応はトークンの値の差し替えだけで実現している。コンポーネント側に @media (prefers-color-scheme: dark) を書いた時点で設計違反。

Dark Tone Before / After

Before: Pure black

背景を完全な黒に置き、本文とカードが強く浮く設定。OLED 感は強いが、長時間読むと目の緊張が残りやすい。

#000000 / #1D1D1F / #D3D3D8

Option: Softened black

背景を少し浮かせ、surface の段差を保ったまま本文の発光感を抑える設定。読み物向けの標準 dark。

#0B0D10 / #202329 / #D0D3D8

Adopted: Raised black

採用値。黒さよりも面の安定感を優先し、長文・コード・表を読むときの緊張を軽くする標準 dark。

使い分け

  • --tint はリンクとアクセントのみ。本文に使わない›
  • ステータス色は意味が固定 — green=Beta/成功、orange=Deprecated/注意、red=Warning/エラー›

Aside（コールアウト）

Apple Docs の aside 語彙そのまま：Note / Important / Tip / Warning / Deprecated。左ボーダー 4px + ごく薄い地色。

注記

背景色のティントは color-mix() でステータス色から導出する。専用 hex を増やさない。

重要

aside は 1 セクションに 1 つまで。並べると警告のインフレが起きて、どれも読まれなくなる。

ヒント

成果物の冒頭に「結論」を aside-tip で置くと、レポートの TL;DR として機能する。

警告

機密情報（API キー・個人情報）を成果物に含めない。公開サーバーに置く前提で書く。

非推奨

旧 ivory + clay テーマの直書き hex は廃止。今後はこのシステムのトークンへ移行する。

要点ボックス（Key Takeaways）

長いレポートの冒頭に置く TL;DR。読者の多くは全文を読まないので、結論と要点をここで先に渡す。地色と罫線は --tint から color-mix() で導出し、生 hex を増やさない。要点は 2〜5 項目に絞る。

要点

  • テキストトークンを WCAG AA（本文 ≥4.5:1）に合わせて微調整した。
  • 見出し↔本文の階層と 1 行の文字数（measure）を読みやすい範囲に整えた。
  • この .summary を追加し、レポート冒頭の要点提示を定型化した。

本文の途中で個別の注意を促すときは Aside を使い、要点ボックスはページ先頭の 1 箇所に留める。両方を多用すると強調が飽和して、どれも読まれなくなる。

コード

フォントはリポジトリ同梱の UDEV Gothic 35LG。コードとテーブル内の mono は 14px を標準にし、ダークでは muted syntax で発光感を抑える（旧 JBM 時代の Medium(500) 持ち上げは、UDEV Gothic に 500 が無いため v0.6.0 で廃止）。コピーボタンは JS が全 pre.listing に自動付与し、ブロックへのホバーでフェード表示 → クリックでコード全文をクリップボードへコピーする（タッチデバイスでは常時表示）。

import SwiftUI

/// プロフィールカードを表示するビュー
struct ProfileCard: View {
    let user: User
    var body: some View {
        VStack(alignment: .leading, spacing: 8) {
            Text(user.name)
                .font(.headline)
            Text("最終ログイン: \(user.lastLogin)")
                .foregroundStyle(.secondary)
        }
        .padding()
    }
}

本文中では --tint や backdrop-filter のようにインラインコードで参照する。

  1. トークンをコピーする。この HTML の :root と dark ブロックを新しいファイルへ。

  3. ブラウザで開いて確認。外観をライト / ダーク両方で見る。

Toshi DS›Patterns›ナビゲーション

ヒント

グローバルナビの背景は color-mix(in srgb, var(--bg) 72%, transparent) + backdrop-filter。スクロールしてもコンテンツがうっすら透ける。

2 カラムレイアウト

詳細ページは「左サイドバー（250px・sticky）+ コンテンツ（最大 760px）」の 2 カラム。ページ内アンカーは右端のフローティングメニューが担当し、グリッドには含めない。

900px 以下ではサイドバーを畳み、1280px 以下ではフローティングメニューを右下のボタン + ポップオーバーに切り替える。

フローティングメニュー

ページ内のセクションへ飛べるアンカーメニュー。広い画面では右端に固定表示（このページの右側に出ているもの）、狭い画面では右下の丸ボタンから開く。

  1. 自動生成。各 .doc-section の data-title から JS がリンクを組み立てる。手書きのメニュー管理はしない。
  2. スクロール連動。IntersectionObserver で現在のセクションを検出し、リンクをハイライトする。
  3. メニュー自体もスクロール可能。max-height + overflow-y: auto で、セクションが多くても破綻しない。

注記

メニューの素材は --sh-menu + blur マテリアル。コンテンツの上に「浮いている」ことを影で示す。

シングルページ

ヘッダーナビもサイドバーもない、コンテンツ + フローティングメニューだけの 1 ページ構成。調査レポートや仕様書を 1 枚で公開するときはこちらを使う。

テンプレートは single-page.html。トークン・UDEV Gothic 35LG・コードのコピー UI・テーマトグルは本体と完全に共通で、 違いはナビゲーション構造だけ。ルーターを持たないため、素のアンカーリンク（#section-id）でセクションへ直リンクできる。

  • single-page.html — テンプレートを開く›

ヒント

新しいレポートを作るときは single-page.html をコピーして、各 .doc-section の id / data-title / 本文を差し替えるだけ。フローティングメニューは自動で追従する。

フッター

薄い区切り線 + Footnote サイズのテキスト。リンクは --text-2、ホバーで下線。

Guidelines

このデザインシステムを成果物に適用するときのルール。迷ったらここに戻る。

原則

  1. コンテンツが主役。装飾は情報の階層を示すためだけに使う。アクセント色は 1 画面 1〜2 箇所。
  2. トークン経由でのみ色を参照。hex 直書きを見つけたらその場で直す。これがダーク対応の生命線。
  3. 単一ファイル + 同梱フォントで完結。外部 CDN への依存なし。読み込んでよいのはリポジトリ同梱の UDEV Gothic 35LG だけ。
  4. Apple の語彙を保つ。eyebrow / abstract / aside / availability — 名前を揃えることで再利用が効く。

新しい成果物での使い方

この design-system.html は「コピペ元」。新しい HTML を作るときは次のブロックから持っていく。

/* 1. :root のトークン一式（light）と
   2. dark の 2 ブロック（@media + [data-theme="dark"]）を必ずセットでコピー */
:root {
  --bg: #FFFFFF;
  --text: #1D1D1F;
  --tint: #0066CC;
  /* …以下トークン一式 */
}

重要

dark ブロックを片方だけコピーすると、手動トグルか OS 追従のどちらかが壊れる。3 ブロック（:root / @media dark / [data-theme=dark]）は常にセットで扱う。

Do / Don't

Do

  • 色は必ず var(--token) で参照する
  • Filled ボタンは 1 画面に 1 つ
  • aside は本当に重要な情報だけに使う
  • セクション見出しに id を振り、アンカー可能にする
  • 公開前にライト / ダーク両方で表示確認する

Don't

  • hex をコンポーネント CSS に直書きしない
  • フォントや JS を CDN から読み込まない（フォントは同梱の UDEV Gothic 35LG のみ）
  • tint 色を本文テキストに使わない
  • aside を連続で 3 つ以上並べない
  • 機密情報・API キーを成果物に含めない
